## Chapter 10: Long-Term Memory System

### 10.1 Design Goals

Maintain persistent memory per chat so the agent can retain user preferences, important facts, and historical context across conversations.

**Key decisions:**
- **Agent-driven**: The LLM autonomously manages memory via the memory tool, with no additional API calls
- **Disk is the source of truth, context is a cache**: All state is stored on disk and read on demand
- **Adaptive token budget injection**: MEMORY.md is always injected, recent daily logs fill the remaining budget in reverse chronological order
- **Substring search**: Zero-dependency grep-like search lets the agent find information across all logs

### 10.2 Storage Structure

```
{memory_dir}/
  SHARED/                    ← Cross-chat shared memory (all allowedUsers can read/write)
    MEMORY.md
  {channel}_{chat_id}/       ← Per-chat isolated memory
    MEMORY.md                ← Long-term curated memory (agent can rewrite to reorganize)
    YYYY-MM-DD.md            ← Daily append-only log
```

**Shared memory layer:** `SHARED/MEMORY.md` stores cross-chat common knowledge (device IPs, network topology, household members, etc.). All chats' agents can read and write it; the path is fixed. Per-chat memory isolation is automatically enforced by `ChatContext` (channel + chat_id) with no additional access control needed.

### 10.3 MemoryStore Interface

```rust
pub struct MemoryStore {
    dir: PathBuf,
    max_memory_bytes: usize,   // Maximum bytes for MEMORY.md (default 4096)
    max_context_bytes: usize,  // Total budget for build_context injection (default 4096)
}
```

| Method | Description |
|--------|-------------|
| `init()` | Create memory_dir if it does not exist |
| `read(ch, cid)` | Read per-chat MEMORY.md; returns empty string if not found |
| `append(ch, cid, text)` | Append to per-chat MEMORY.md; returns a warning if max_memory_bytes is exceeded |
| `rewrite(ch, cid, text)` | Fully rewrite per-chat MEMORY.md (used by agent to reorganize) |
| `read_shared()` | Read SHARED/MEMORY.md; returns empty string if not found |
| `append_shared(text)` | Append to SHARED/MEMORY.md; returns a warning if max_memory_bytes is exceeded |
| `rewrite_shared(text)` | Fully rewrite SHARED/MEMORY.md |
| `read_log(ch, cid, date)` | Read the log for a given date; returns empty string if not found |
| `append_log(ch, cid, text)` | Append to today's `YYYY-MM-DD.md` |
| `search(ch, cid, query, max)` | Substring search across all per-chat files, ±2 lines of context, in reverse chronological order |
| `search_shared(query, max)` | Search SHARED/MEMORY.md |
| `build_context(ch, cid)` | Adaptive token budget injection (see 10.4) |

### 10.4 Adaptive Token Budget Algorithm

`build_context()` generates the memory context injected into the system prompt:

1. Inject SHARED/MEMORY.md first (shared memory takes priority as cross-chat common knowledge)
2. Then inject per-chat MEMORY.md (ensures long-term memory is preserved)
3. Fill remaining budget with recent daily logs in reverse chronological order
4. For each daily log file: include it in full if it fits; skip it entirely if it does not (no truncation)
5. Use `### Shared MEMORY.md` / `### MEMORY.md` / `#### YYYY-MM-DD` as section headings

If shared memory is empty, the `### Shared MEMORY.md` heading is omitted.
If MEMORY.md itself exceeds the budget, it is truncated to the budget (the on-disk file is not affected).

### 10.5 MemoryTool (6 actions)

Tool description injected into the LLM:

```
Manage persistent memory across conversations.
- read: Read your long-term MEMORY.md
- append: Add new information to MEMORY.md
- rewrite: Reorganize and rewrite MEMORY.md (use when it's getting full)
- read_log: Read a daily log entry (defaults to today)
- append_log: Append to today's daily log
- search: Search across all memory files for a keyword or phrase

Use scope: "shared" for cross-conversation knowledge (device IPs, network config,
household members, universal preferences). Default scope is per-conversation.
```

Tool schema:
```json
{
  "action": "read | append | rewrite | read_log | append_log | search",
  "content": "string (for append/rewrite/append_log)",
  "date": "string YYYY-MM-DD (for read_log, optional — defaults today)",
  "query": "string (for search)",
  "scope": "string enum ['chat', 'shared'] (optional, default 'chat')"
}
```

### 10.6 System Prompt Injection

Structure produced by `build_system_prompt(base, tools, memory_context, compaction_warning)`:

```
{base_prompt}

## Current Date & Time
Thursday, February 19, 2026 — 14:30

## Memory
{memory_context — generated by MemoryStore::build_context()}

## Available Tools
- ...

{compaction_warning — appended only when messages.len() > 15}
```

**Date/time injection:** Dynamically generated using `chrono::Local::now()`, formatted as `"%A, %B %e, %Y — %H:%M"` (e.g. `Thursday, February 19, 2026 — 14:30`). The agent knows the current time without needing to call a get_time tool.

**Compaction warning text:**
```
⚠ Conversation history is approaching its limit and older messages will be
lost. Store any important context, decisions, or user preferences to memory
now (action: "append" or "append_log"). Reply with the user's answer
afterward. If nothing to store, just reply normally.
```

### 10.7 Memory Overhead

| Component | Resident heap |
|-----------|---------------|
| `MemoryStore` | ~48 B (PathBuf + 2×usize) |
| `Arc<Mutex<ChatContext>>` | ~64 B |
| Reading MEMORY.md / build_context | Temporary ≤max_context_bytes, freed after request |
| search operation | Temporary: files read one by one + matching lines collected, freed after request |
| **Total** | **< 200 B resident** |

## Chapter 11: Cron Automatic Execution System

### 11.1 CronTask Data Structure

```rust
pub struct CronTask {
    pub id: String,                      // First 8 characters of a UUID v4
    pub cron_expr: String,               // 5-field cron expression
    pub description: String,             // Human-readable description
    pub command: String,                 // Command / instruction to execute
    pub channel: String,                 // Channel at creation time (e.g. "telegram")
    pub chat_id: String,                 // Chat ID at creation time
    pub created_at: i64,                 // Creation timestamp
    pub last_run: Option<i64>,           // Last execution timestamp (minute-level dedup)
    pub schedule_at: Option<String>,     // ISO 8601 one-shot trigger time
    pub delete_after_run: bool,          // Self-delete after execution
    pub delivery_mode: String,           // "announce"|"webhook"|"none"
    pub webhook_url: Option<String>,     // Webhook delivery URL
    pub isolated: bool,                  // Isolated execution (uses a temporary session)
}
```

`channel` and `chat_id` are automatically injected from `Arc<Mutex<ChatContext>>` when a task is added. All new fields use `#[serde(default)]` to ensure backward-compatible deserialization of existing data.

### 11.2 cron_matches — 5-Field Matching

```rust
pub fn cron_matches(expr: &str, now: &chrono::DateTime<chrono::Local>) -> bool
pub fn schedule_at_matches(schedule_at: &str, now: &chrono::DateTime<chrono::Local>) -> bool
```

**cron_matches**: 5-field `min hour dom mon dow`. Supports `*` (wildcard), numeric values, comma-separated lists, ranges (`1-5`), steps (`*/5`), and range+step (`1-30/2`).

**schedule_at_matches**: Accepts RFC 3339 (`2026-02-18T10:30:00+08:00`) or NaiveDateTime (`2026-02-18 10:30:00`); compares against the current minute.

### 11.3 Main Loop Execution Flow

A 60-second interval branch is added to the `tokio::select!` in `main.rs`:

```
Every 60s tick:
  1. load_all_tasks(cron_file)
  2. for each task:
     - Minute-level dedup: last_run/60 == now/60 → skip
     - Determine trigger: check schedule_at first → otherwise check cron_expr
     - Isolated execution: isolated=true → use temporary chat_id `_cron_isolated_{id}_{ts}`
     - Update chat_context → construct IncomingMessage → agent.handle()
     - Delivery dispatch:
       ├─ "announce" → send_text to original channel
       ├─ "webhook" → SSRF check then POST JSON to webhook_url (5s timeout)
       └─ "none" → silently discard
     - task.last_run = now.timestamp()
     - delete_after_run → mark for deletion
  3. Remove marked tasks
  4. if changed: save_all_tasks(cron_file)
```

**Minute-level dedup:** `last_run / 60 == now_timestamp / 60` ensures a task is not executed more than once within the same minute.

**Skip first tick:** `cron_interval.tick().await` consumes the initial immediate trigger before the loop starts, preventing accidental execution at startup.

**Webhook SSRF protection:** `validate_url_ssrf()` is called before delivery to validate the target address.

### 11.4 Memory Overhead

~50 bytes per task (new fields). Reuses the main loop's tokio timer (`tokio::time::interval`); cron.json is only read temporarily during each per-minute check.

## Chapter 12: Extension Capabilities — exec, web_search, and MCP Client

### 12.1 exec Tool

A shell command execution tool. Spawns a child process via `sh -c` and supports:

- Configurable timeout (default 30 s, maximum 300 s)
- Separate capture of stdout and stderr; excessively long output is automatically truncated
- Skills directory is automatically prepended to PATH, supporting reusable saved scripts
- **Environment variable isolation**: `env_clear()` clears all inherited environment variables, exposing only `PATH`, `HOME=/app`, `TERM=dumb`, and `LANG=C.UTF-8`. This prevents the LLM from reading API keys, bot tokens, and other process secrets via `env`/`printenv`

**Config:** `ExecConfig { timeout_secs, max_output_bytes, work_dir, skills_dir }`

**File:** `src/tools/exec.rs` (~160 lines)

### 12.2 web_search Tool

A web search tool with dual search-engine support:

**DuckDuckGo (default):**
- Requests `https://html.duckduckgo.com/html/?q={query}`
- Parses HTML with pure string matching; no HTML parser dependency

**Brave Search (optional):**
- GET `https://api.search.brave.com/res/v1/web/search?q={query}`
- JSON API, `X-Subscription-Token` authentication
- Free tier: 2 000 requests/month
- Automatically falls back to DuckDuckGo if Brave request fails

**Config:** `WebSearchConfig { provider: "duckduckgo"|"brave", api_key_env: "BRAVE_SEARCH_API_KEY", max_results: 5 }`

**File:** `src/tools/web_search.rs` (~250 lines)

### 12.3 MCP Client

A Model Context Protocol client that lets users integrate any MCP server without modifying Rust code:

- stdio transport: newline-delimited JSON-RPC 2.0
- Only four methods implemented: `initialize`, `notifications/initialized`, `tools/list`, `tools/call`
- Each MCP server tool is registered in ToolRegistry under the name `mcp__{server}__{tool}`
- Graceful shutdown: close stdin → wait 2 s → kill

**Resource limits (matching Claude Code):**
- `timeout_secs` (default 60): per-`tools/call` timeout, `tokio::time::timeout()` wraps `recv()`
- `max_output_bytes` (default 65536): tool output is truncated to this many bytes (UTF-8-safe truncation) with a truncation notice appended
- `truncate_mcp_output()` + `safe_truncate_pos()` helper functions ensure truncation never occurs in the middle of a multi-byte character

**Config:**
```json5
{
  "mcp": {
    "servers": {
      "weather": {
        "command": "python3",
        "args": ["server.py"],
        "env": { "API_KEY": "${WEATHER_API_KEY}" },
        "timeoutSecs": 60,
        "maxOutputBytes": 65536
      }
    }
  }
}
```

**Memory overhead:** ~16 KB per server (BufReader/BufWriter) + ~200 B per tool

**File:** `src/tools/mcp.rs` (~300 lines)

## Chapter 13: Anthropic OAuth 2.0 Authentication

### 13.1 Design Goals

Support Anthropic OAuth (PKCE flow) authentication so that personal account (Claude Max/Pro) OAuth tokens can be used to call the API without needing a separately provisioned API Key. Fully backward-compatible with the existing API Key mode.

### 13.2 Authentication Modes

```rust
pub enum AuthMode {
    ApiKey(String),                          // Traditional x-api-key header
    OAuth(Arc<Mutex<TokenStore>>),           // Bearer token + automatic refresh
}
```

Set `auth.mode` to `"api_key"` (default) or `"oauth"` in config to switch modes.

### 13.3 OAuth PKCE Flow

```
User /auth → generate code_verifier + challenge(S256) → return authorization URL
User authorizes in browser → receive authorization code
User /auth CODE → POST token_url to exchange for token → save to file
Subsequent API calls → Bearer token + anthropic-beta header → auto-refresh before expiry
```

**Key parameters:**

| Parameter | Value |
|-----------|-------|
| Auth URL | `https://claude.ai/oauth/authorize` (Max/Pro subscription) |
| Token URL | `https://console.anthropic.com/v1/oauth/token` |
| Callback | `https://console.anthropic.com/oauth/code/callback` |
| Client ID | Configurable; default `9d1c250a-e61b-44d9-88ed-5944d1962f5e` |
| Scopes | `org:create_api_key user:profile user:inference` |
| PKCE | S256 (SHA-256 + base64url-no-pad) |
| state | Random 32-byte base64url-encoded value (CSRF protection) |
| code | `true` (required query parameter) |
| API Header | `Authorization: Bearer <token>` + `anthropic-beta: oauth-2025-04-20` |

### 13.4 TokenStore

```rust
pub struct TokenStore {
    client_id: String,
    token_url: String,
    file_path: PathBuf,
    tokens: Option<TokenData>,
    pkce: Option<PkceState>,      // Temporary state while authorization is in progress
    http_client: reqwest::Client,
}

pub struct TokenData {
    access_token: String,
    refresh_token: String,
    expires_at: i64,              // Unix timestamp (seconds)
}
```

- `get_token()` — returns a valid token, automatically refreshing 5 minutes before expiry
- `load()` / `save()` — JSON file persistence
- Token file path is configurable (default `./auth_tokens.json`)

### 13.5 ClaudeProvider Dual Authentication

The `ClaudeProvider.auth` field replaces the original `api_key`:

- `ApiKey` → `x-api-key: <key>` header (existing behavior)
- `OAuth` → `Authorization: Bearer <token>` + `anthropic-beta: oauth-2025-04-20`

Headers are set uniformly via the `apply_auth()` method, shared by both `chat()` and `chat_stream()`.

### 13.6 /auth Command

The `/auth` command is intercepted on the Telegram side:

| Command | Behavior |
|---------|----------|
| `/auth` | Generate PKCE, return authorization URL |
| `/auth CODE` | Exchange authorization code for token |
| `/auth status` | Show authentication status |
| `/auth reset` | Clear tokens |

### 13.7 Configuration

```json5
{
  "auth": {
    "mode": "oauth",                    // "api_key" (default) | "oauth"
    "clientId": "9d1c250a-e61b-44d9-88ed-5944d1962f5e",
    "tokenFile": "./auth_tokens.json"
  }
}
```

### 13.8 Practical Notes

**API Endpoint:** OAuth tokens still use `api.anthropic.com/v1/messages`, the same endpoint as API Key mode.

**`anthropic-beta` header is required:** Omitting `anthropic-beta: oauth-2025-04-20` returns
`authentication_error: OAuth authentication is currently not supported`. This header must be
included to enable OAuth authentication.

**Use short model IDs:** In OAuth mode, some date-suffixed model IDs (e.g. `claude-opus-4-6-20250610`,
`claude-sonnet-4-5-20250514`) may return 404 not_found. Use the short names without a date suffix:
- `claude-opus-4-6` (recommended)
- `claude-sonnet-4-5`

**Authorization URL construction:** Always use `url::Url::query_pairs_mut()` to correctly encode
all parameters. Manual `format!` concatenation can cause parameters to be dropped (e.g. the
`Missing client_id parameter` error).

**Required authorization URL parameters:**
- `code=true` — required; omitting it causes an unknown error
- `state=<random>` — required; omitting it returns "Missing state parameter"
- `client_id` — required; encoding errors return "Missing client_id parameter"

**Token reuse:** The OAuth token used by the Claude Code CLI (`~/.claude/.credentials.json`) can
be reused directly. The format is `sk-ant-oat01-...` (access) and `sk-ant-ort01-...` (refresh).
Note that Claude Code tokens may include the `user:sessions:claude_code` scope, which differs
slightly from tokens obtained via an independent OAuth flow.

**Token file mount:** When deploying with Docker, the token file must be mounted with read-write
access (not `:ro`), because automatic token refresh updates the file contents.

**Token exchange must use JSON:** The POST request body for token exchange must be
`application/json`; `application/x-www-form-urlencoded` is not accepted and returns
`invalid_request_error: Invalid request format`. The request body must also include the `state`
parameter.

**Telegram Markdown escaping:** The client_id in the authorization URL contains `_` characters,
which Telegram's Markdown mode interprets as italic markers, causing the URL to be truncated.
Escape `_` as `\_` before sending the URL.

**Auth code may contain URL fragment:** The auth code copied from the callback page by the user
may include a `#state_value` suffix (URL fragment). The code must automatically strip everything
from `#` onward; otherwise the server returns `invalid_grant: Invalid 'code' in request`.
